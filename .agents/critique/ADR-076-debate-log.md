## architect (system design, boundaries, coherence with ADR-073/ADR-072/ADR-066/ADR-035): APPROVE-WITH-CHANGES

Design is sound and ships as Phase 1. Separation of concerns is clean: the SHA gate stays the single hard safety boundary, the lease is an advisory ergonomics hint, and the ADR documents the line explicitly so no reader confuses which mechanism protects the branch. Fail-open is the correct default here and does not contradict ADR-066: ADR-066 mandates fail-closed-and-loud for safety hooks, whereas this is a coordination layer sitting on top of an already-hard safety gate, so a store outage degrading to today's behavior is strictly correct. The sub-second acquire race (no atomic CAS on PR comments) is disclosed honestly and absorbed by the unchanged SHA gate; the lease never claims mutual exclusion. Security treatment is rigorous: lease comment is treated as untrusted input, strict anchored parse, MAX_TTL enforced at parse time to bound forgery DoS, forgeable signal kept away from the authorization decision (CWE-78 and CWE-918 surfaces named). Quality-attribute trade is explicit (collision-reduction and timeline-legibility bought; true mutual exclusion sacrificed). Phased rollout is reversible; Phase 1 delivers value with no remote dependency and Phase 2 is gated behind a BOT_PAT audit. Top two-year failure mode: dual representation drift as future loops invent ad-hoc coordination and the advisory-vs-authoritative line blurs; the ADR mitigates by fixing the owner vocabulary and naming the SHA gate as the sole hard boundary. Two residual notes, non-blocking: (1) the 15-minute TTL is asserted without latency evidence from real autofix cycles, acceptable as a fixed legible constant with a future-ADR escape hatch already stated; (2) the draft-staging-copy rename note (lines 12-18) is a process hazard around the BLOCKING architect PreToolUse gate, ensure the rename and debate run in the main checkout so the gate sees the evidence.

## critic (completeness, ambiguity, unstated assumptions; canonical-source verification against actual files): APPROVE-WITH-CHANGES

Design judgment is sound and I want to be clear the must_fix items are accuracy/specification defects, not a rejection of the approach. The advisory-lease-beside-hard-SHA-gate shape is correct, fail-open is the right default (consistent with ADR-066), the 15-min TTL self-heal is reasonable, and the security section (advisory-not-authorization, MAX_TTL parse enforcement, no auto-fetch) is genuinely good and blocks the forged-lease-DoS escalation path. The alternatives table honestly rejects git-notes and external lock stores.

Three softer concerns the implementer should weigh, not gates: (1) Cost/latency unstated. Decision part 3 says acquire does 'one read' of head.sha AND the lease, but those are two different GitHub surfaces (PR head via GraphQL, comments via a list call). Scanning ALL comments on a hot PR to find the latest marker is unbounded as comment count grows; .claude/rules/release-it.md requires an explicit timeout on this integration point and the ADR does not name one. State the timeout and a bound on comments scanned. (2) The phased rollout leans entirely on Phase 1 delivering value, but Phase 1's only real coordination case is two LOCAL pr-autofix sessions on the same branch simultaneously, which the ADR never establishes as a frequency that justifies the machinery (#2611 was local-vs-remote, a Phase 2 case). The success metric ('collision rate drops measurably') has no baseline measurement defined, so 'measurably' is untestable as written; name the metric source and the pre-change baseline. That is in Consequences>Negative already, good, but quantify against expected autofix cadence.

All four cited related ADRs (035, 066, 009, 014) exist. The ADR ships no code, so the blast radius of accepting it is bounded to the recorded contract; the must_fix corrections are cheap to make in the document before the canonical rename.

## independent-thinker (contrarian: is the lease the right mechanism, or is there a simpler/cheaper approach?): APPROVE-WITH-CHANGES

DESIGN IS SOUND, accept it. The ADR is unusually intellectually honest: it admits the lease is advisory, has no atomic compare-and-set, cannot eliminate the sub-second race, and keeps the SHA gate as the only hard safety boundary. That honesty is what makes it pass the hardest contrarian probe. Fail-open is correct and mandatory (a fail-closed lease would convert a comment-store outage into a workflow outage, strictly worse than the conflict). TTL self-expiry as the liveness mechanism is the right call: no heartbeat protocol, a crashed holder cannot wedge the branch. The Security section is genuinely good: it treats the lease comment as untrusted input, enforces MAX_TTL at parse time to bound forgery DoS, and correctly reasons that a forged lease can at most cause a self-DoS (peer SKIPs) and never an overwrite because the SHA gate is independent of the lease. That is the right threat model.\n\nWHAT I CHALLENGED HARDEST (the contrarian core): the ADR proves the lease is A working mechanism but not that it is the CHEAPEST mechanism that clears the bar. The delivered value reduces to 'detect the collision before the fix work instead of at push time'. A pre-fix re-fetch of head.sha (re-running the existing SHA check earlier in the loop) captures most of that for the loser-detects case with no new store, no parser, no injection surface. The lease's genuine edge over that is it lets the SECOND loop see the FIRST loop is active before either starts duplicate work, which a pure SHA recheck cannot (SHA only advances after the first loop pushes). That edge is real but the ADR never states it as the load-bearing justification; it should, because that single sentence is the whole reason the lease beats the cheaper option. Make that the headline of the alternatives analysis.\n\nSCOPE/VALUE concern: the committed, shippable slice (Phase 1, local-only) addresses the rare case (two concurrent local sessions). The common, expensive case (local-vs-remote-CI, the actual #2611 incident) is Phase 2, gated behind an external audit that may never land. The ADR is reversible and additive, so this is acceptable, but the value timing should be stated honestly rather than implied to ship with Phase 1.\n\nThe four must-fix items are all cheap (two are doc-accuracy, one is an alternatives-table addition, one is a one-sentence scope honesty statement). None is a substantive flaw that stops acceptance. Verify them in or before Phase 1 implementation; do not block the ADR. This is APPROVE-WITH-CHANGES, not BLOCK: the mechanism is safe, reversible, and correctly subordinate to the SHA gate; the changes sharpen the justification and fix references, they do not rescue a broken design.

The lease's real delivered value is moving collision detection from push-time to pre-fix-time. You can get most of that by re-fetching head.sha immediately before starting fix work and aborting/re-syncing if it advanced, with zero comment-store machinery, zero parser, zero injection surface. The table jumps from 'status-quo push-time SHA gate' straight to 'comment lease' and never weighs this cheaper midpoint. Either show why pre-fix-recheck is insufficient (e.g. it does not prevent BOTH loops starting, only detects after the fact) or adopt it as the Phase 1 floor under the lease. ## security: APPROVE-WITH-CHANGES

Architecture is sound. The core security move is correct: the lease is advisory and fail-open, and the Force-Push Safety SHA gate (real git SHAs, independent of the lease) remains the only push authorization. Forgery is therefore isolated to self-DoS on the autofix loop, never an overwrite. The Security section already anticipates CWE-78/CWE-502 (strict anchored parse, fixed owner vocabulary, 40-hex base_sha, no auto-fetch, never executed) and the acquire race (CWE-362) is honestly framed as not providing mutual exclusion. Threat scores: forgery Low (3/10), stale-lock DoS Low (3/10 after MAX_TTL fix), injection Low (2/10 if parser built as specified), acquire race Low (2/10). The three must_fix items are parser-contract gaps, not design flaws: must_fix 1 closes a real bypass of the ADR's own stated TTL-forgery bound; must_fix 2 closes an identity-trust gap between verified comment author and forgeable body fields; must_fix 3 caps scan cost. All belong in the Security section before Phase 1 implements the parser, so the implementer cannot get the trust boundary subtly wrong. None block acceptance because the SHA gate is independent of all three. Phase 2's BOT_PAT permissions audit is correctly gated. Note: the draft staging path and rename-after-debate workflow are out of scope for the security lens but the file move should happen as the ADR instructs. Phase 1 pytest matrix in Implementation Notes should add a negative case for each must_fix (far-future-acquired_at forgery returns 'no live lease'; forged owner/session by a non-authoring credential is not treated as self-renewal; oversized comment flood is bounded).

must_fix:
- MAX_TTL must anchor to the reader's clock, not the forgeable acquired_at. Acquire/parse step rejects any marker whose expires_at > now + 15m (now = reader's clock). The ADR's relative check (expires_at > acquired_at + MAX_TTL) is bypassable: a forger sets acquired_at and expires_at 15 min apart but both far-future (e.g. 2030-01-01), passing the relative check while reading as live indefinitely. The stated 'self-DoS bounded to one TTL' guarantee only holds with a now-anchored bound (CWE-400 / CWE-367).
- Self-renewal and self-edit decisions MUST key on the GitHub comment author (the verified credential that posted the comment), NOT on the body's self-declared owner/session strings, which are forgeable. State this in part 1 and acquire step 3. As written, a forger writing 'owner: local:pr-autofix, session: session-2587' could be treated as the legitimate holder's own lease and edited/trusted; the owner/session body fields are traceability/display only (CWE-345 origin validation).
- State an explicit upper bound on the lease-comment timeline scan (e.g. latest-N items, matching check_pr_live_state.py's existing scan window) so a PR flooded with forged <!-- PR-AUTOFIX-LEASE --> comments cannot become an unbounded parse-cost DoS on the acquire path (CWE-400).

## analyst: APPROVE-WITH-CHANGES

The protocol design is sound. The SHA gate remains the sole hard safety boundary and the ADR never relaxes it. Fail-open on store error is correctly specified and consistent with ADR-066. The ACT/SKIP pattern reuse from check_pr_live_state.py is validated by the existing code. The session-ID linkage to new_session_log.py is implementable. Phase 2 is correctly gated behind a BOT_PAT audit with no premature commitment. The three must-fix items are specification gaps, not design flaws. The SHA gate absorbs the safety consequence of all three gaps, so none creates a safety regression. They will produce inconsistent or unverifiable implementations if not resolved before Phase 1 code ships. Resolve in an ADR amendment or in the Phase 1 PR description before the implementer writes scripts/pr_autofix_lease.py. Two additional open questions are not blocking: (1) the BOT_PAT audit acceptance criteria and auditor role are unspecified, acceptable for Phase 0 but must be resolved before Phase 2 scoping; (2) the owner vocabulary extension governance path is unspecified ('plus any future loop' without a change process).

must_fix:
- Tombstone evaluation rule is ambiguous. Part 1 states 'the tombstone's expires_at being in the past is sufficient to signal the lock is free' but a tombstone with expires_at in the future would be read differently by two implementers. Replace with an unambiguous rule: a comment with owner: none is always free regardless of expires_at. A live lease requires both owner not none AND expires_at in the future. This is a specification defect, not a safety issue (SHA gate absorbs it), but it will produce inconsistent implementations if unresolved before scripts/pr_autofix_lease.py is written.
- Unknown-but-valid owner value is unspecified. The acquire algorithm treats malformed (unparseable) leases as 'no live lease' and overwrites them. It does not specify behavior for a syntactically valid lease with an unrecognized owner value (e.g., owner: unknown-bot). Specify explicitly: either treat unrecognized-but-valid owner as a live lease (conservative, safe) or as no live lease (fail-open). Either is defensible; unspecified produces divergent implementations across Phase 1 and Phase 2.
- Success metric 'collision rate drops measurably' is not currently measurable. No existing signal in session logs or SHA gate output captures push-blocked-by-SHA-gate as a distinct, queryable event. The baseline before Phase 1 cannot be established. Before Phase 1 ships, either instrument the SHA gate to emit a structured lease_collision_blocked event (establishing a baseline), or amend the success metric to one measurable from existing signals such as 'zero force-push overwrites observed in SHA gate output across N monitored sessions'.
